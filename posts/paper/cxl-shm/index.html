0、背景知识

DSM（Distributed Shared Memory）:分布式共享内存的核心思想是将物理上分散的内存空间虚拟化，使得不同机器上的进程能够像访问本地内存一样访问远程节点上的内存。

  • 通过软件的支持，将不同节点的内存合并成一个统一的逻辑内存空间。这样，程序员可以直接访问分布在各个节点上的内存，就像访问本地内存一样。
  • 为了确保不同节点之间对共享内存的访问是正确的，DSM系统需要确保内存一致性。

Hazard Era算法：是一种主要应用于并发编程中的内存管理算法，尤其是在无锁数据结构和并发垃圾回收方面，常用于多线程环境中管理内存回收。 它的核心思想是通过 “危险指针”（hazard pointers）来避免内存泄漏和悬空指针（dangling pointer）的问题，同时支持高效的并发内存回收。

  1. 危险指针（Hazard Pointers）：允许线程标记它们正在访问的对象，这样其他线程就可以知道某个对象当前被访问，不能被回收， 使得其他线程不会错误地访问已被删除或正在删除的数据。
  2. 危害时期（Hazard Era）：为了防止在并发环境中出现垃圾回收（GC）误操作（例如一个线程正在访问对象，但另一个线程已经删除了该对象）， Hazard Era 引入了时期（era）的概念。每个线程持有一个“当前时期”（current era），它表示线程最近操作的内存周期。在某个时期内， 某个线程可能会访问多个对象，并且所有对象的生命周期都会被标记为属于该时期。当回收内存时，只有在确认没有线程在访问某个对象时，才会回收它。
  • RPC(Remote Procedure Call，远程过程调用) 是一种计算机通信协议，它使得一个程序能够调用另一个程序（通常运行在不同的计算机上）的函数或过程，就像调用本地函数一样。传统的传值(Pass by value)调用方将参数值通过序列化（Serialization）转换成可以跨网络传输的格式，传递给远程的被调用方。返回值也需要反序列化。这带来了序列化和网络开销。
  • 幂等操作（Idempotent Operation）是指无论操作被执行一次还是多次，产生的结果都是相同的(保证一致性)。这种特性对于分布式系统和容错设计尤为重要，因为在网络故障或系统崩溃的情况下，可能会重试某些操作。

1、问题背景和解决方案

论文中提出分布式共享内存（DSM）系统在部分失败（例如客户端或机器崩溃）的情况下的内存管理问题。传统DSM系统通常需要程序员手动管理内存生命周期，这对复杂的应用场景（如微服务和分布式系统）来说是一大挑战，因为这种情况下可能会导致内存泄漏、双重释放和野指针问题。此外，现有的容错方法通常是阻塞式的，难以应对延迟敏感的请求。

本文提出的 CXL-SHM 是一个基于引用计数的自动化分布式内存管理系统，用于解决分布式共享内存（DSM）在部分失败场景下的内存管理难题。其设计特点有

  • 内存管理：通过一种基于时代的非阻塞算法，解决了因客户端崩溃导致的内存泄漏、双重释放和野指针问题。
  • 容错能力：能够在参与客户端发生部分失败（如进程或机器崩溃）时正常运行，无需阻塞其他客户端。

2、CXL-SHM overview

接口

cxl_malloc：创建一个 root_ref()，用于容忍故障。支持部分失败的容忍，它不是线程安全的，需要CXL-SHM 提供的 API 传递引用。 cxl_send_to: 显式地将 CXLRef 对象发送到另一个线程
csl_receive_from: 接收 CXLRef 对象，这个过程会确保引用计数被正确地更新，从而维护内存对象的生命周期

内存布局

Arena：整个共享内存池的顶层结构，负责管理所有的内存段。
Segment：在 arena 中分配的独占内存区域，每个线程会从arena中分配一个独占的segment。
Page：每个 segment 进一步划分为多个页，每个页用于分配特定大小类的内存对象。
Block：每个页内包含多个固定大小的内存块，用于实际的内存对象分配

每个页内的内存块以链表的形式组织，页的元数据中包含指向第一个空闲块的指针。这样，分配和释放内存块的操作可以在页内进行

部分故障弹性自动内存管理

1. void AttachReference(Client client, Addr *ref, Block *refed) {
2. do {
   1. saved = refed->header;
   2. saw_cid, saw_era = saved.lcid, saved.lera;
   3. if (Era[client.cid][saw_cid] < saw_era)      // 更新 Era 矩阵，标记 client “看见”了上一个客户端的提交，同步各个客户端之间的操作状态
   4.      Era[client.cid][saw_cid] = saw_era;
   5. cur_era = Era[client.cid][client.cid];
   6. client.redo = {Attach,cur_era,ref,refed,saved.ref_cnt};      // 执行重做日志，客户端本地进行
   7. newh = Header(client.cid, cur_era, saved.ref_cnt+1);         // 修改头部，事务提交点 Modifyrefcnt 是判断重做的关键
   8.  } while(!refed->header.compare_and_exchange(saved, newh));  // 原子指令确保只有一个并发线程会执行
   9.  *ref = refed;        // Modifyref
   10. Era[client.cid][client.cid]++;
3.  }

这是 CXL-SHM 中的分布式引用计数维护中的引用附加操作，每一个客户端通过目标对象的当前头部更新状态，使得所有相关联的客户端能够同步信息，并且把本地提交的信息更新到 header 中。那么如果部分客户端出现故障的情况呢？

分为故障检测和故障恢复执行两个步骤：

故障检测

崩溃后的恢复由一个独立的恢复服务对每一个客户端的重做日志(Redo Log)进行检测，根据操作类型 attach，当前时代，目标引用指针和对象，当前读取到的引用计数来判断哪个客户端出现了崩溃。

故障恢复执行

恢复服务检查以下两个条件，判断是否需要重做操作：

  • 条件 1：目标对象的头部 lcid 和 lera 与崩溃客户端的重做日志匹配。
    如果匹配，说明 ModifyRefCnt 已执行，但 ModifyRef 未完成，需要重做 ModifyRef。
  • 条件 2：比较客户端的本地时代与其他客户端所观察到的时代。
    如果 Era[i][i] <= max(Era[j][i])（其中 j ≠ i），说明其他客户端观察到了崩溃客户端的 ModifyRefCnt 操作，因此也需要重做 ModifyRef。

注意：

  1. 每个引用指针应该只由一个编写器拥有（并因此被修改）。这种单写入器多读取器并发模型确保在写入器失败后不会修改引用的值，因此是ModifyRef的幂等性。
  2. ModifyRefCnt 的执行需要保证一次性成功，一旦成功，不可再重做。故使用源自指令 CAS
  3. 根据事务的 幂等性 划分为两个阶段，通过 Modifyrefcnt 提交状态可以判断事务是否完成

CXL-SHM 的实现

CXL-SHM 内存分配CXLobj 顺序应该是分配 RootRef(分布式引用的起点，用于记录本地引用的状态，并存储指向实际数据对象（CXLObj）的偏移指针)、分配实际数据对象(CXLobj 包含lcid、lera)、建立 RootRef 与 CXLObj 之间的链接、CAS 增加引用计数。顺序不能够改变，考虑到在任何环节客户端崩溃的情况，如果先分配了 cxlobj，在连接的过程崩溃，会导致 rootref 没有指向的实际内存块，内存泄漏；反之先链接，分配数据块崩溃，那么就会出现野指针。

CXL-SHM 使用了两层引用计数机制：

  1. 在同一线程内，克隆或销毁一个对象的引用只会修改 RootRef 对象的 ref_cnt，不需要使用原子指令或刷新操作，仅在本地修改引用计数
  2. 跨进程或跨机器，会创建新的 RootRef 对象，这意味着引用计数会在 CXLObj 的头部进行更新，而这些更新通常会被原子操作（例如 CAS）所封装

实验

环境和测试

  • 硬件技术：CXL技术，特别是 CXL 3.0 版本，它支持内存共享。
  • 测试平台：由两个计算节点组成的CXL基础的RDSM（可部分故障恢复的分布式共享内存）测试平台，共享一个外部CXL内存设备。
  • 微基准测试：用于评估 CXL-SHM 的效率，包括内存分配和释放操作的性能。
  • 端到端应用：通过实际应用来展示 CXL-SHM 构建高效分布式应用的简单性和灵活性，例如基于引用传递的RPC框架和共享一切的分布式键值存储。

表现

  • 与现有技术比较：与mimalloc和jemalloc等现代易失性内存分配器相比，CXL-SHM的性能大约低一个数量级，这主要是由于CXL-SHM需要额外的内存栅栏和缓存刷新操作。
  • 恢复性能：CXL-SHM的恢复过程能够以每秒约2350万个对象的速度从客户端故障中恢复，远快于基于垃圾收集的持久内存分配器。
  • RPC框架：CXL-RPC 的性能比基于 RDMA 的 RPC 高出3.83至4.62倍，因为它避免了序列化、反序列化和网络传输的开销。
